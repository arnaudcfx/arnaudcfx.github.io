INFO0027 – Project 2: File Explorer

Java Design Patterns OOP Swing GUI

Software Architecture

The File Explorer project focuses on implementing an extensible, object-oriented architecture using multiple design patterns to handle various filesystem-like entities. Each entity—root, folder, file, copy, alias, and archive—inherits from a common abstract class FileExporterItem. This unified inheritance allows consistent management of user interactions and entity-specific behaviors through the Visitor pattern.

Core functionalities include recursive entity cloning, alias referencing, and file compression through the Deflater class, implemented via specialized visitor classes such as CreateArchiveVisitor, CompressItemVisitor, CreateCopyVisitor, and DisplayVisitor.

Design Patterns

Creational Pattern – Singleton

The Singleton pattern is applied to the Root class to ensure that only one root instance exists across all graphical interfaces. This guarantees global access and consistency within the File Explorer.

  • Advantages: Single root instance, accessible across the system.
  • Drawbacks: May violate the Single Responsibility Principle.

Structural Patterns

Composite Pattern

The Composite pattern models the hierarchical tree structure of the File Explorer, allowing unified treatment of entities (files or folders). Each subclass of FileExporterItem implements behavior tailored to its type.

  • Advantages: Enables polymorphism, recursion, and extension without code breaking.
  • Drawbacks: General base class may include unused or null-returning methods.

Proxy Pattern

The Proxy pattern is employed for the alias entity, providing indirect control and reference to a real file. This facilitates multiple alias instances pointing to the same file while managing content retrieval.

  • Advantages: Enables control over remote or shared files; supports multiple aliases.
  • Drawbacks: Requires cleanup if the original file is deleted.

Decorator Pattern

The Decorator pattern is used in the copy operation to wrap and recursively clone entities. It enhances object behavior dynamically without altering the original interface.

  • Advantages: Supports deep cloning, respects SRP, allows feature extension.
  • Drawbacks: Dependent on other entities’ clone() behavior.

Behavioral Pattern – Visitor

The Visitor pattern provides a mechanism for performing operations such as creating aliases, archives, copies, and displaying entities. Each visitor subclass defines custom behavior for each entity type (File, Folder, Root, etc.) via methods like visitFile() or visitFolder().

  • Advantages: Decouples operations from entities, avoids redundancy, respects Open/Closed Principle.
  • Drawbacks: Generic implementation required to separate GUI and logic layers.
